# 从 0 到开源维护者的 90 天计划

这份计划适合还没有成熟开源项目，但希望逐步建立真实维护记录的人。

## 第 1-7 天：建立项目基础

- 创建公开 GitHub 仓库
- 设置 GitHub profile 为 public
- 完成 README、LICENSE、CONTRIBUTING、SECURITY
- 添加 issue template 和 PR template
- 添加基础 CI
- 发布第一个可用版本或文档集

验收标准：

```text
陌生开发者能在 5 分钟内理解项目用途
知道如何贡献
知道如何报告问题
能看到维护者和路线图
```

## 第 8-30 天：做真实维护

- 每周至少处理一次 issue 或文档更新
- 把常见问题整理进 FAQ 或模板
- 发布至少 1 个 release
- 修复读者反馈中的错误
- 补充 2-3 个可复用示例

验收标准：

```text
仓库有连续 commit 记录
有 issue 讨论或 PR
有 changelog
有 release 记录
```

## 第 31-60 天：扩大可用性

- 邀请开发者试用模板
- 收集真实反馈
- 增加标签体系和 triage 规则
- 增加维护者周检查清单
- 把模板拆成更细的场景

验收标准：

```text
至少有一批真实用户反馈
模板能被复制到实际项目中使用
issue 不只是自问自答
```

## 第 61-90 天：形成维护证据

- 建立稳定 release 节奏
- 处理外部贡献
- 总结项目价值和维护职责
- 整理 stars、forks、用户反馈、下载或引用数据
- 准备开源维护支持类申请材料

验收标准：

```text
你能证明自己是 primary maintainer
仓库有持续维护记录
项目对某类开发者有明确价值
Codex/API credits 的用途能落到具体维护工作
```

## 不建议做的事

- 为了制造活跃度提交无意义 commit
- 批量创建没有用户价值的 issue
- 把 README 写得很大但没有内容
- 夸大用户、下载量或生态影响
- 把 fork 项目伪装成原创项目
