# 团队导入 AI 工具指南

这份指南帮助小团队把 AI 开发工具引入真实工程流程，重点关注权限、代码隐私、review 责任、试点范围和验收指标。

## 适用场景

- 团队准备试用 ChatGPT、Codex、Copilot、Claude、Gemini 等 AI 开发工具。
- 团队已有代码仓库、PR 流程和 CI。
- 团队希望先试点，而不是一次性替换现有流程。

## 导入前检查

| 检查项 | 需要确认的问题 |
| --- | --- |
| 代码权限 | AI 工具是否会访问私有仓库、内部文档或客户数据？ |
| 数据边界 | 哪些内容不能粘贴到 AI 工具？ |
| 责任归属 | AI 生成的代码由谁 review，谁负责上线结果？ |
| 使用范围 | 先用于文档、测试、PR review，还是允许修改业务代码？ |
| 审计记录 | 是否保留关键提示词、输出和人工判断？ |
| 回滚机制 | AI 生成的改动出错后如何回滚？ |

## 推荐试点范围

第一阶段建议只选择低风险流程：

- 文档草稿
- README 改进
- issue triage
- release note 草稿
- 测试用例建议
- CI 报错初步分析

暂不建议在第一阶段直接让 AI 处理：

- 生产配置
- 权限系统
- 密钥和凭证
- 数据迁移
- 支付、风控、安全策略
- 大规模自动重构

## 角色分工

| 角色 | 职责 |
| --- | --- |
| 维护者 | 决定 AI 工具可以进入哪些流程，最终确认合并结果 |
| Reviewer | 检查 AI 输出是否符合项目规范和安全边界 |
| 使用者 | 按模板提供上下文，不粘贴敏感信息 |
| 负责人 | 追踪试点指标，决定是否扩大范围 |

## 试点流程模板

```text
目标：
在 2 周内验证 AI 工具是否能降低 PR review、issue triage 和文档维护成本。

范围：
- 允许：文档、测试建议、CI 日志分析、release note 草稿
- 不允许：生产配置、密钥、权限策略、客户数据

记录：
- 每次使用保留任务类型、输入摘要、输出摘要和人工判断
- 对错误建议记录原因

验收指标：
- PR review 初稿时间是否减少
- issue 分类是否更稳定
- CI 排错是否更快定位到原因
- 是否出现安全或隐私风险
```

## 验收指标

建议每周记录：

- AI 参与的 PR 数
- AI 参与的 issue 数
- 节省的估计时间
- 被采纳的建议数量
- 被拒绝的建议数量
- 错误建议的主要类型
- 是否出现敏感信息风险

## 退出条件

如果出现以下情况，应该暂停试点：

- 成员频繁粘贴敏感信息
- AI 生成内容未经 review 直接合并
- 错误建议导致生产事故
- 团队无法判断 AI 输出是否正确
- 工具使用成本高于维护收益
