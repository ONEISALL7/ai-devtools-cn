# AI 工具接入工程流程

这份文档描述如何把 AI 开发工具接入真实工程工作，而不是只用于一次性生成代码。

## 核心原则

AI 工具应该进入已有工程流程，而不是替代工程流程。一个可靠的使用方式通常包含：

```text
问题定义 -> 上下文整理 -> AI 执行 -> 人类 review -> 测试验证 -> 记录决策 -> 合并发布
```

## 适合使用 AI 的维护任务

- 梳理 issue 背后的真实问题
- 起草 PR review 初稿
- 补充单元测试和回归测试
- 分析失败的 CI 日志
- 重构小范围重复代码
- 生成 release note 草稿
- 改进 README、CONTRIBUTING 和 API 文档
- 整理安全升级或依赖升级影响

## 不适合完全交给 AI 的任务

- 未经验证直接合并代码
- 处理未知来源的密钥和生产数据
- 做安全判断但没有人工复核
- 大规模重构但没有测试保护
- 引入新依赖但没有维护成本评估

## 推荐工作流

### 1. 明确任务边界

在使用 AI 前，先写清楚：

```text
目标是什么
哪些文件可以改
哪些行为不能变
如何验证结果
是否需要兼容旧版本
```

### 2. 提供足够上下文

高质量上下文通常包括：

```text
项目结构
相关文件
失败日志
已有测试
用户反馈
约束条件
期望输出格式
```

### 3. 要求 AI 输出可 review 的结果

不要只要求“修好它”。更好的要求是：

```text
请先说明原因，再给出最小修改方案。
修改后列出验证步骤。
如果需要新增依赖，请说明理由和替代方案。
```

### 4. 人工 review

维护者需要重点检查：

```text
是否改变公开行为
是否破坏兼容性
是否引入安全风险
是否缺少测试
是否隐藏复杂度
是否符合项目风格
```

### 5. 记录维护结果

每次重要维护动作至少保留：

```text
issue 或 PR 链接
变更原因
验证方式
后续风险
release note 条目
```

这些记录会成为项目长期维护价值的公开证据。
