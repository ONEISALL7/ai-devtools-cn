# Python PR Review 案例：API 调用约束变更

## 适用场景

这是一个可复制的 Python 项目审查示例，适合维护者快速给出可执行 review 建议。

## 背景

一个 PR 改动了公开函数签名，并在多个调用点仅更新了部分调用路径。

## 可复制输入

```text
请 review 这个 PR：

背景：这是一个 Python 工具库，新增了 `fetch_users(org: str, *, timeout: int)`。

变更：
- 函数定义新增 keyword-only 参数 timeout。
- 某些调用点未更新，新增了默认值 None。
- 新增了一个测试但断言仍基于旧签名。

请输出：
1. 是否阻塞项
2. 建议修复清单
3. 安全/兼容性风险
4. 可直接放进 PR comment 的文本
5. 最小验证命令
```

## 预期输出示例

```text
结论：Comment

阻塞项：
- 主要函数签名变更未同步所有调用点，CI 运行时将出现 TypeError 风险。

建议修复：
- 使用关键词参数方式统一更新所有调用点。
- 为新增参数补充默认值测试与边界测试。
- 在现有测试中补充对 None/超范围参数的行为断言。

风险：
- 与外部下游调用兼容性受影响。
- 新增参数缺少类型约束将放大 runtime 风险。

验证命令：
- python -m pytest
- python -m pip check

PR 评论草稿：
当前改动已实现核心功能，但仍有未覆盖调用点。建议先补齐 `fetch_users` 在全部调用路径的更新，并增加最小回归测试，避免运行时签名异常。
```

## 可复用价值

- 可直接作为 PR review 任务模板。
- 可用于 issue triage：区分阻塞修复与可迭代后续。
- 可转化为教学案例，放在 onboarding 的“审查要点”中。
