# 依赖升级风险评估案例：主版本升级决策模板

## 适用场景

当项目准备升级核心依赖（如开发工具、CI 插件、lint 规则包）到新主版本时，判断是否能合并。

## 可复制输入

```text
请评估以下依赖升级是否应立即合并：

项目背景：Node.js 开源仓库，CI 依赖 markdownlint、eslint。

当前版本：markdownlint 0.31.0
目标版本：markdownlint 0.33.0

已知风险：
- 变更日志提到默认配置变更
- 某些规则 ID 下线
- 新版要求 Node 版本门槛提升

约束：
- 不要只给“升级建议”，要给风险分层和验证边界。
- 给出回滚路径和 release note 提示。

请输出：
- 风险清单（高/中/低）
- 验证范围
- 回滚方案
- 是否建议立即 merge
```

## 期望输出示例

```text
结论：有条件推进。

高风险：
- 变更日志显示规则行为调整，可能影响当前规则集合。
- Node.js 最低版本提升会影响仍用旧版本 runner 的开发者。

中风险：
- 仅文档行为变化，对大多数仓库无直接影响。

低风险：
- 配置文件可兼容迁移，有可回退方案。

验证：
1. 在本地 Node 版本矩阵上运行 lint。
2. 对比关键文件在旧/新配置下的结果差异。
3. 更新 CI 缓存 Key，避免老版本 node_modules 残留影响。

回滚：
- 以 PR 级别保留 lockfile 可复现点。
- 如发现大面积规则误报，立即回退到 0.31.x，并关闭本次合并。

Release notes 提示：
- 标注 Node 版本门槛与迁移注意事项，提示受影响环境。
```

## 维护者执行标准

- 是否能直接写进 review comment。
- 是否包含“是否可 merge”与“何时回退”。
- 是否明确要求验证命令。

## 可复用边界

- 项目内部依赖较敏感时优先只合并升级 PR 到单独分支并全量验证。
- 不将“可选迁移”写成“必须立即合并”。

## 使用后产物

- `dependency-upgrade-risk-template.md` 的输入材料
- review comment
- 变更说明与 release note 条目
- 风险决策的内部知识资产
