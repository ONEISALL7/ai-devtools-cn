# 公开安全脱敏模板

这个模板用于把准备公开到 issue、PR、反馈、案例、release note 或社区帖子的内容先做安全检查和脱敏。它适合处理 CI 日志、错误栈、配置片段、用户反馈摘要和案例草稿。

它不能替代人工安全审查。任何涉及真实客户、生产事故、未公开源码或账号凭证的内容，都应该由维护者再次确认。

## 复制使用

````text
你是一个公开安全脱敏助手。请帮我检查下面准备公开发布的内容，找出不能公开的信息，并给出可公开替代表达。

## 准备公开的场景

- 发布位置：GitHub issue / PR / release note / 文档 / 社区帖子 / 其他
- 内容类型：CI 日志 / 错误栈 / 配置片段 / 用户反馈 / 案例草稿 / 代码 diff / 其他
- 是否来自公开仓库：
- 是否涉及真实用户、客户、公司或生产环境：
- 是否需要保留技术细节用于复现：

## 原始内容

```text
在这里粘贴准备公开的内容。粘贴前请先手动删除明显的 token、密码和私钥。
```

## 请检查这些风险

1. 凭证和密钥：API key、token、cookie、私钥、连接串、验证码、session。
2. 内部基础设施：内部域名、IP、仓库地址、服务名、队列名、数据库名。
3. 客户和个人信息：姓名、邮箱、手机号、订单、合同、公司名、用户 ID。
4. 未公开源码：私有函数、业务逻辑、内部 diff、专有配置。
5. 生产事故细节：真实故障时间线、影响范围、补救细节、值班人员。
6. 账号和权限：用户名、组织名、权限组、访问策略、云资源 ID。

## 需要你输出

请按以下结构输出：

1. 不能公开的内容：列出具体片段和风险类型。
2. 建议替换方式：给出可公开的抽象描述。
3. 可保留的技术细节：说明哪些信息可以保留以便复现问题。
4. 脱敏后版本：输出一版可以公开的文本。
5. 仍需人工确认：列出 AI 无法判断、必须由维护者确认的内容。
6. 发布建议：可以公开 / 脱敏后公开 / 不建议公开。

## 限制

- 不要编造不存在的上下文。
- 不要为了保留细节而暴露敏感信息。
- 如果不确定某个字段是否敏感，默认建议替换成占位符。
- 占位符要可读，例如 `<internal-service>`、`<user-email>`、`<api-token>`。
- 输出要方便直接复制回 GitHub issue、PR 或文档。
````

## 常用占位符

| 原始类型 | 建议占位符 |
| --- | --- |
| API key / token | `<api-token>` |
| cookie / session | `<session-cookie>` |
| 私有域名 | `<internal-domain>` |
| 内部服务名 | `<internal-service>` |
| 客户名称 | `<customer-name>` |
| 用户邮箱 | `<user-email>` |
| 订单或合同 | `<customer-record>` |
| 私有仓库 | `<private-repository>` |
| 生产环境 ID | `<production-resource>` |

## 人工复核清单

- [ ] 没有 token、密码、cookie、私钥或连接串。
- [ ] 没有客户、用户、同事或公司的可识别信息。
- [ ] 没有内部域名、IP、私有仓库或生产资源 ID。
- [ ] 没有未公开源码、内部业务规则或专有配置。
- [ ] 脱敏后仍保留足够技术信息，能让读者理解问题。
- [ ] 如果来自他人反馈，已获得公开或匿名整理许可。
