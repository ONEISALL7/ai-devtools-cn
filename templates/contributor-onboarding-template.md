# 外部贡献者引导模板

用途：帮助开源维护者把第一个可完成的贡献任务讲清楚，降低外部贡献者提交 PR 的门槛。

适用场景：

- 你有 `good first issue`，但新贡献者不知道怎么开始。
- 你想邀请外部开发者贡献文档、示例、测试或小修复。
- 你需要把贡献步骤、验收标准和安全边界写成一段可复制说明。

## 输入信息

```text
项目名称：
仓库地址：
适合新贡献者的 issue：
建议贡献类型：文档 / 示例 / 测试 / 小 bug fix / 模板补充
技术栈或文档栈：
本地运行或验证命令：
PR 需要包含的内容：
不应该改动的范围：
维护者希望收到的反馈：
```

## 提示词正文

你是一个开源项目维护者助手。请基于我提供的信息，写一份给外部贡献者的贡献引导说明。

要求：

1. 先用 3 句话说明这个项目解决什么问题、这个 issue 为什么适合第一次贡献。
2. 给出从 fork、建分支、修改、验证到提交 PR 的步骤。
3. 把任务拆成 3-6 个可检查的小步骤。
4. 写清楚 PR 验收标准，包括必须运行的命令、文档要求和不应改动的范围。
5. 给出一段 PR 描述模板，方便贡献者直接复制。
6. 给出维护者回复草稿，用于欢迎贡献者并说明 review 预期。
7. 明确安全边界：不要提交密钥、token、客户数据、内部日志、未公开源码或个人隐私。
8. 不要夸大项目影响力，不要承诺一定合并 PR。

## 输出格式

```markdown
# 贡献引导

## 适合谁

## 你要完成什么

## 开始前

## 操作步骤

## 验证命令

## PR 验收标准

## PR 描述模板

## 维护者回复草稿

## 安全提醒
```

## 验收标准

- 新贡献者能在 10 分钟内判断自己是否适合参与。
- 任务范围足够小，不需要理解整个项目。
- 验证命令明确。
- PR 描述模板能直接复制。
- 安全提醒不会要求贡献者公开敏感信息。

## 注意事项

- 不要把泛泛的愿景写成贡献任务。
- 不要把“帮我完善项目”写成 issue，需要给出具体文件、示例或验证命令。
- 如果 issue 涉及安全、账号、生产数据或私有系统，不适合作为外部首次贡献任务。
